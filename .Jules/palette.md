## 2026-04-24 - Screen Reader Friendly Form Validation
**Learning:** When custom CSS like `.sr-only` is not permitted, using the `aria-label` attribute directly on input elements (`<input>`, `<textarea>`, `<select>`) provides the exact same context for screen readers without adding extra DOM nodes or requiring new styles. Additionally, `role="alert"` works excellently for dynamic form errors. For loading and asynchronous states that use conditional rendering (like `*ngIf`), `aria-live="polite"` must be applied to a *persistent* wrapper container (that always exists in the DOM) rather than on the conditional elements themselves; otherwise, screen readers may fail to announce the change when the element appears.
**Action:** Use `aria-label` on inputs if visual labels are omitted and custom CSS is prohibited. Always use `role="alert"` for dynamically displayed validation messages so they are immediately announced. Apply `aria-live="polite"` to persistent wrapper elements, not dynamically rendered elements, for reliable announcements of loading and result states.
